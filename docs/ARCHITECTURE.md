# 抽签小系统 - 架构设计

## 1. 需求摘要

- **模式**：支持随机抽取 / 内定结果
- **流程**：两轮抽取
  - **第一轮**：从全部候选者中抽取 N 家（N 可配置，如 5–10）
  - **第二轮**：从第一轮结果中抽取 1 家为最终结果

---

## 2. 核心概念

| 概念 | 说明 |
|------|------|
| 候选池 | 参与抽签的全体名单（如企业/单位列表） |
| 第一轮名额 | 第一轮要抽出的数量，可配置（如 5–10） |
| 第一轮结果 | 第一轮抽出的 N 家 |
| 最终结果 | 第二轮从第一轮结果中抽出的 1 家 |
| 抽签模式 | 随机 / 内定（实现层面支持内定，但 UI 不显式暴露「内定」概念） |

---

## 4. 领域模型（核心实体与值对象）

### 4.1 实体

- **Candidate（候选者）**  
  - 属性：id、名称、可选扩展信息  
  - 表示候选池中的一条记录  

- **Draw（一次抽签）**  
  - 属性：id、第一轮名额、模式（随机/内定）、内定结果 id（可选）、创建时间  
  - 行为：执行两轮，产出第一轮结果 + 最终结果  

### 4.2 值对象 / 聚合结果

- **Round1Result**：第一轮结果（N 个 Candidate）
- **FinalResult**：最终结果（1 个 Candidate）
- **DrawConfig（实现简化后可选）**：若需要，可作为一次抽签的输入配置（第一轮名额、模式参数等）；当前实现不再持久化「全局抽签设置」，而是每次调用 `/api/draw` 时直接传入第一轮名额与模式。

### 4.3 抽签策略（接口 + 实现）

- **IDrawStrategy**（或类似命名）  
  - `drawFrom(candidates, count)` → 从给定列表中按规则选出 count 个  

- **RandomDrawStrategy**  
  - 使用随机数从 candidates 中无放回抽取 count 个  

- **FixedDrawStrategy**  
  - 内定：在当前实现中，若启用内定模式且未显式传入内定 id，则**使用候选池中 id 最小的一条记录**作为内定结果；第一轮在包含该候选的前提下随机补足其余 N-1 个，第二轮直接返回该候选。  

策略由应用层根据「本次抽签模式」选择，领域层不关心存储与 UI。

---

## 5. 核心流程：一次完整抽签

1. **输入**  
   - 候选池（列表，来自当前底库）  
   - 第一轮名额 N（5–10 的整数）  
   - 模式（随机/内定）  
     - 随机模式：前端普通点击「执行第一轮抽签」按钮即触发（不传或传 `random`）  
     - 内定模式：前端通过 **按住 Shift 再点击** 同一按钮触发（传 `fixed`），不会在 UI 上出现「内定」相关文案  
   - 若内定且需要覆盖默认行为，可选地传入内定候选 id（当前 UI 未暴露此能力，后端接口保留参数）

2. **校验**  
   - 候选池非空；N 在 [min, max] 内（如 5–10）。  
   - 内定模式下：  
     - 若显式传入内定 id，则校验其存在于候选池中；  
     - 若未传，则使用候选池中 id 最小的一条作为内定对象（由后端按导入顺序 / id 排序自动选择）。  

3. **第一轮**  
   - 使用当前策略从**全体候选池**中抽取 N 家 → 得到 **Round1Result**。  
   - 内定模式：第一轮结果中必须包含内定候选（可先选内定，再随机补足 N-1）。  

4. **第二轮**  
   - 从 **Round1Result** 中抽取 1 家 → 得到 **FinalResult**。  
   - 随机模式：随机选 1 个；内定模式：直接取内定候选。  

5. **输出与持久化**  
   - 返回 Round1Result + FinalResult；  
   - 将本次 Draw 的模式、第一轮名额、内定候选（若有）及两轮结果持久化到 SQLite 中，用于历史查询与审计。  

---

## 6. 数据流示意

```
候选池 ──(第一轮，抽 N 个)──► 第一轮结果 ──(第二轮，抽 1 个)──► 最终结果
         ↑                                    ↑
         │                                    │
    随机：无放回随机选 N 个              随机：从 N 个中随机选 1 个
    内定：含内定者 + 随机补足 N-1        内定：直接为内定者

当前实现中，「内定者」默认为候选池中 id 最小的一条记录，前端通过 Shift+点击同一抽签按钮触发内定模式，界面文案不暴露「内定」字样。
```

---

## 7. 配置与扩展点

- **第一轮名额**：可配置最小值、最大值（如 5–10），当前实现为用户在抽签页输入一个 5–10 范围内的整数。  
- **随机源**：抽象为接口（如 `IRandomProvider`），便于测试时注入固定种子或伪随机，生产可接系统随机。  
- **存储**：当前实现采用 SQLite 单文件存储候选池与抽签历史；若后续需要，也可以替换为其它 DB，而不改领域与用例。  
- **审计**：若需要公正性，可为每次 Draw 记录时间、配置、两轮结果、随机种子（若允许保存）。

## 8. 技术选型建议（候选规模约 200 家）

**规模结论**：200 家数据量很小，内存/计算均无压力，选型主要看**使用场景**（单机 vs 多端、是否要留痕审计），不必为性能选重方案。

### 推荐优先级

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| **现场单机抽签、少留痕** | **纯前端（Vue/React + localStorage/JSON）** | 一个 HTML 或 SPA 即可，无需部署服务；200 家列表可内嵌或从本地 JSON 加载；随机用 `crypto.getRandomValues` 即可。 |
| **需要历史记录、多设备或审计** | **Node + SQLite（或 JSON 文件）** | 抽签与记录在服务端，数据一致、可查历史；SQLite 单文件、零配置，足够 200 家 + 大量历史记录。 |
| **仅命令行、现场快速用** | **Node/Python CLI** | 脚本读候选 CSV/JSON，参数指定第一轮数量、是否内定；输出两轮结果到终端或文件，实现最快。 |

### 不推荐

- **重型后端 + MySQL/PostgreSQL**：200 家无必要，增加部署与维护成本。
- **仅依赖 `Math.random()`**：若需一定公正性，建议浏览器端用 `crypto.getRandomValues`、Node 用 `crypto.randomInt`。

---

## 9. 小结

- **两轮固定**：第一轮从总数抽 N 家，第二轮从 N 家中抽 1 家。  
- **模式分离**：随机 vs 内定通过策略模式实现，应用层组装配置与策略。  
- **隐式内定**：实现上支持内定模式，默认以内存中 id 最小的候选为内定对象，通过前端 Shift+点击触发，但 UI 不显式暴露「内定」概念。  
- **分层清晰**：展示 → 应用 → 领域 → 基础设施，后续加存储、审计、多端都容易扩展。
